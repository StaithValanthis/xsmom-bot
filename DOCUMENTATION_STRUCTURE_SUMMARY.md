4. **KB Update Mechanism** - `tools/update_kb.py` auto-generates module maps and config references
5. **Complete Content** - All critical docs populated with real content (not just skeletons)

## Knowledge Base (KB)

### Purpose

The KB is the **authoritative map** of xsmom-bot's framework:
- **Current state**: What the bot **is today**
- **Design rationale**: **Why** design choices were made
- **Change history**: **How** the framework evolved

### Structure

**Core Documents:**
1. **`kb/knowledge_base.md`** - KB entrypoint and index
2. **`kb/framework_overview.md`** - Complete framework map (â­ start here)
3. **`kb/change_log_architecture.md`** - Framework-level changes over time
4. **`kb/autogenerated/`** - Auto-generated summaries (module maps, config refs)

### KB Maintenance

**Auto-Generated Content:**
- Module Map (`docs/architecture/module_map.md`) - From `src/` scan
- Config Reference (`docs/reference/config_reference.md`) - From `config.yaml` parse

**To regenerate:**
```bash
python -m tools.update_kb
```

**When to regenerate:**
- After adding new modules to `src/`
- After modifying `config/config.yaml.example`
- Before committing major changes (consider pre-commit hook)

**Manual Updates:**
- `kb/framework_overview.md` - Major architectural changes
- `kb/change_log_architecture.md` - Framework-level changes

See [`docs/meta/style_guide.md`](docs/meta/style_guide.md) for KB maintenance guidelines.

## KB Update Tool

### Tool: `tools/update_kb.py`

**Purpose:**
- Auto-generate module maps from `src/` scan
- Auto-generate config reference from `config.yaml` parse
- Update KB timestamps

**Usage:**
```bash
# Update all KB docs
python -m tools.update_kb

# Only update module map
python -m tools.update_kb --skip-config-ref

# Only update config reference
python -m tools.update_kb --skip-module-map
```

**Outputs:**
- `docs/architecture/module_map.md` - Module tree with descriptions
- `docs/reference/config_reference.md` - Parameter-by-parameter reference
- `docs/kb/knowledge_base.md` - KB timestamp updated

**Integration:**
- Can be added to pre-commit hook (optional)
- Can be run manually before committing
- Can be run in CI pipeline (optional)

## Maintenance Workflow

### When Framework Changes

1. **Make framework change** (code/config change)
2. **Update `kb/framework_overview.md`** - Reflect current state
3. **Add entry to `kb/change_log_architecture.md`** - Document change
4. **Regenerate auto-generated docs** - `python -m tools.update_kb`
5. **Update related docs** - Architecture docs, usage guides, etc.
6. **Commit changes** - Include KB updates in same commit

### When Config Changes

1. **Make config change** (add/modify parameters)
2. **Update `config/config.yaml.example`** - Keep example current
3. **Regenerate config reference** - `python -m tools.update_kb --skip-module-map`
4. **Update `architecture/config_system.md`** - If config system changes
5. **Commit changes** - Include config ref update

### When Adding New Modules

1. **Add new module** to `src/`
2. **Regenerate module map** - `python -m tools.update_kb --skip-config-ref`
3. **Update `kb/framework_overview.md`** - If framework changes
4. **Update `architecture/high_level_architecture.md`** - If architecture changes
5. **Commit changes** - Include module map update

## Documentation Philosophy

### Clarity First

- **Be clear and direct** - Avoid jargon unless necessary
- **Be concise** - Get to the point quickly
- **Be practical** - Focus on actionable information

### Current and Opinionated

- **Current** - Reflect what the bot **is today**, not months ago
- **Opinionated** - Explain **why** design choices were made, not just what they are

### Organized and Cross-Linked

- **Organized** - Clear structure, logical grouping, easy navigation
- **Cross-linked** - References relevant docs, easy to navigate

### Easy to Maintain

- **Auto-generated** - Module maps and config refs auto-generated from code
- **Clear workflow** - When to update what
- **Simple tooling** - `tools/update_kb.py` makes updates easy
